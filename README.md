Security Operations Center Laboratory Guide and Threat Hunting Roadmap
Introduction to the Controlled Security Operations Architecture
The deployment of a modern Security Operations Center (SOC) laboratory requires a meticulous orchestration of distinct operating systems, telemetry agents, network monitors, and log analytics platforms. The objective of this comprehensive roadmap is to outline the construction, operationalization, and threat-hunting workflows within an isolated, controlled network environment.1 This environment consists of a Kali Linux instance serving as the primary adversarial simulation node, a Metasploitable virtual machine acting as a highly vulnerable Linux target, a Windows endpoint representing a standard corporate workstation, and an Ubuntu server functioning as the central hub for security monitoring and Security Information and Event Management (SIEM) operations.1 Photographic evidence generated within this environment identifies the Kali Linux attack node operating on the IP address 192.168.56.10, while the Metasploitable target operates on the IP address 192.168.56.103 over the enp0s8 host-only network interface.
By progressing through a fourteen-day operational roadmap, the infrastructure matures from bare-metal setup to a highly tuned detection engine capable of identifying complex adversarial behaviors. The roadmap dictates that each core technology or phase is operated and optimized over a strict two-day lifecycle. This deliberate pacing ensures that telemetry generation, log ingestion, alert tuning, and dashboard visualization are fully realized before introducing subsequent variables.1 The architecture facilitates the collection of endpoint logs, the monitoring of suspicious activity, the detection of attacker behavior, and the investigation of alerts.1
The Windows endpoint serves as the primary log source, heavily augmented by Microsoft Sysinternals Sysmon to provide deep telemetry that standard Windows Event Logs fail to capture adequately.1 The Ubuntu server hosts the Splunk Enterprise SIEM, Wazuh for Host-based Intrusion Detection (HIDS), and Suricata for Network Intrusion Detection (NIDS).1 The adversarial infrastructure is driven by Kali Linux, utilizing tools such as Nmap for network reconnaissance, Hydra for brute-force attacks, Metasploit for exploitation, Burp Suite for web application testing, and Wireshark alongside terminal monitors like htop and vtop for deep system and packet-level analysis.1
Phase 1: Infrastructure Deployment and SIEM Initialization (Days 1-2)
Establishing the Centralized Logging Pipeline
The first two days of the laboratory lifecycle are dedicated to establishing the foundational logging pipeline, fulfilling the prerequisite for SOC creation. The core of this pipeline is Splunk Enterprise, installed natively on the Ubuntu monitoring server.1 Splunk functions as the central data repository, search engine, and dashboarding platform. The Splunk web interface is made accessible via http://localhost:8000, while the backend management daemon, splunkd, operates on port 8089 to handle application programming interface (API) communication, configuration management, and the indexing pipeline.1
A critical component of this initial deployment phase is the integration of Sysmon on the Windows endpoint. Advanced persistent threats and localized attackers frequently utilize Living Off the Land Binaries (LOLBins)—trusted, built-in Windows tools such as powershell.exe, rundll32.exe, and certutil.exe—to evade traditional signature-based antivirus detection mechanisms.1 Standard Windows event logging mechanisms often miss the nuanced command-line arguments passed to these executables, creating a dangerous visibility gap. Sysmon bridges this gap by hooking deep into the operating system kernel to log high-fidelity events such as process creations (Event ID 1), network connections (Event ID 3), and domain name system (DNS) queries (Event ID 22).1 The installation of Sysmon requires executing the command sysmon64.exe -accepteula -i sysmonconfig.xml via an administrative command prompt.1 The -accepteula flag ensures the installation proceeds without graphical manual prompts, while the -i flag installs Sysmon as a persistent Windows service using the specified XML configuration file, which defines the precise telemetry parameters and inclusion/exclusion logic to be captured.1 Once installed and operational, Sysmon hooks into telemetry providers, captures process events, and writes them to the Windows Event Viewer under the specific path of Applications and Services Logs -> Microsoft -> Windows -> Sysmon -> Operational.1
Resolving Telemetry Ingestion and Parsing Errors
During the initial log ingestion phase across Days 1 and 2, several critical errors commonly arise that require immediate remediation and explanation to ensure the integrity of the SOC monitor. The first significant error involves the failure of the Splunk SIEM to return any results when a SOC analyst queries specific numerical Event IDs, such as executing EventID=1 or EventID=3 in the search bar.1 The root cause of this operational failure lies in the raw Extensible Markup Language (XML) structure of the Sysmon logs being forwarded to the indexer.1 Splunk receives the full XML data blob but fails to automatically extract nested XML tags into structured, searchable key-value fields.1 To rectify this parsing error, regular expression (regex) extraction must be applied manually within the Search Processing Language (SPL) query.1 The command | rex field=_raw "<EventID>(?<EventID>\d+)</EventID>" is appended to the base search.1 This SPL command instructs the Splunk processing engine to search the raw unformatted log field (_raw), locate the literal <EventID> XML tags, and capture the numerical digits contained within them, creating a new, permanently searchable field called EventID.1 Similar manual rex extraction commands are required to successfully parse the CommandLine, Image, DestinationIp, and DestinationPort fields out of the raw XML structure.1
The second major ingestion error manifests when the analyst attempts to query the standard Windows Security logs using the query index=main source="WinEventLog:Security", which results in a persistent "No results found" error message.1 This absence of data indicates a failure in the Splunk Universal Forwarder's local configuration to monitor standard authentication events, which are absolutely crucial for detecting brute-force attacks and lateral movement (specifically Event Code 4625 for failed logons and 4624 for successful logons).1 The fix requires manual modification of the Splunk inputs configuration file located at the absolute path C:\Program Files\Splunk\etc\system\local\inputs.conf on the Windows endpoint.1 The file must be updated via a text editor to include the `` stanza.1 Within this stanza, the analyst must define disabled=0 to explicitly enable the input, index=main to route the collected logs to the correct primary index, and current_only=0 to ensure that both historical logs present before the installation and future logs are read by the forwarder.1 Following these manual configuration file modifications, the Splunk forwarder service must be restarted using the command splunk restart executed from within the C:\Program Files\Splunk\bin directory to
apply the new ingestion parameters.1
Phase 2: Network Reconnaissance and Packet Monitoring (Days 3-4)
Operating Nmap for Surface Mapping
With the defensive logging infrastructure stabilized and parsing correctly, Days 3 and 4 introduce the first offensive tool into the laboratory environment: Nmap (Network Mapper). Nmap is executed from the Kali Linux instance (operating on IP 192.168.56.10) to map the attack surface of the Metasploitable virtual machine (operating on IP 192.168.56.103).5 The Metasploitable architecture is intentionally engineered with numerous legacy vulnerabilities and misconfigurations to facilitate advanced penetration testing and SOC detection exercises.3
The reconnaissance phase begins with the execution of a stealthy Transmission Control Protocol (TCP) Synchronize (SYN) scan combined with aggressive service version detection. The specific command nmap -sS -sV -O 192.168.56.103 is utilized.3 The -sS parameter initiates the half-open SYN scan, which is historically quieter than a full TCP connect scan.3 In a SYN scan, the Nmap engine sends a SYN packet to the target port. If a SYN-ACK packet is received, the port is deemed open, and Nmap immediately sends a Reset (RST) packet to tear down the connection before the final ACK packet is sent, preventing the target application from fully logging a completed connection.3 The -sV parameter probes open ports to determine the exact service daemon and version running by comparing banner responses against the nmap-service-probes database, while the -O parameter attempts to fingerprint the underlying operating system by analyzing the specific nuances of the target's TCP/IP stack implementation.3
The Nmap scan results reveal a critical finding on TCP port 21: the presence of the vsftpd 2.3.4 service.3 The File Transfer Protocol (FTP) is inherently insecure when transmitting data in plaintext, but this specific version of the Very Secure FTP Daemon (vsftpd) harbors a highly documented and intentional supply-chain vulnerability.3 Recognizing this vulnerable service version dictates the exact trajectory of the subsequent exploitation phase and the specific network detection rules that must be authored within the SIEM.6

-sS	Executes a TCP SYN (Stealth) scan by sending a SYN packet and responding with an RST upon receiving a SYN-ACK, leaving the connection half-open to evade rudimentary application logs.
-sV	Enables rigorous service version detection by sending application-specific probes to open ports and comparing the responses against a comprehensive signature database.
-O	Enables aggressive Operating System detection by analyzing TCP/IP stack implementation details such as initial sequence number generation, IP ID sequencing, and window sizes.
<target-ip>	The destination IPv4 address, explicitly identified as 192.168.56.103 within the controlled laboratory environment.

Establishing the Wireshark Packet Capture Baseline
Simultaneous to the execution of the Nmap reconnaissance, the SOC analyst operates Wireshark on the Ubuntu monitoring server to capture the network traffic flowing across the enp0s8 host-only adapter. Wireshark operates in promiscuous mode, capturing all frames transmitted between the Kali Linux attacker (192.168.56.10) and the Metasploitable victim (192.168.56.103). The capture interface displays the raw TCP three-way handshake initiated by the Nmap SYN scan, illustrating the rapid transmission of SYN packets across all 1000 common ports. This packet-level visibility is critical for engineering highly specific Network Intrusion Detection System (NIDS) rules, allowing the analyst to define exactly what a port scan looks like at the byte level before moving into active exploitation. The captured traffic serves as the baseline of normal versus reconnaissance network behavior, providing the foundation for subsequent Suricata signature development.
Phase 3: Exploitation Frameworks and Vulnerability Troubleshooting (Days 5-6)
Exploiting the vsftpd 2.3.4 Backdoor via Metasploit
Days 5 and 6 transition the laboratory operations from passive reconnaissance to active, weaponized exploitation using the Metasploit Framework. The vulnerability discovered in vsftpd 2.3.4 (CVE-2011-2523) is a malicious backdoor that was secretly introduced into the official downloadable source code archive between June 30 and July 3, 2011.7 The vulnerability is triggered by a specific sequence of bytes during the standard authentication process. If a client attempts to log in with a username containing a smiley face character sequence :), the vulnerable vsf_sysutil_extra() C function is executed by the daemon.6 This malicious function completely bypasses normal authentication checks and executes execl("/bin/sh","sh",(char *)0);, effectively spawning a root-level command shell bound to TCP port 6200.9 The password provided during this interaction is entirely irrelevant and is discarded by the underlying code.12
To execute this exploit, the analyst launches the Metasploit console by typing msfconsole in the Kali Linux terminal.13 Once the framework initializes, the specific module is loaded into memory using the command use exploit/unix/ftp/vsftpd_234_backdoor.3 The analyst must configure the environmental variables required by the module to target the correct asset. Executing show options reveals that the RHOSTS (Remote Host) variable must be explicitly defined.13 The target IP is specified using set RHOST 192.168.56.103.13 The default payload for this specific exploit is cmd/unix/interact, which simply commands Metasploit to connect to the newly opened port 6200 socket upon successful exploitation.10 The attack is initiated with the command run.13
However, photographic evidence from the laboratory execution reveals a critical failure in the exploitation chain. The output of the Metasploit console displays:
[*] 192.168.56.103:21 - Banner: 220 (vsFTPd 2.3.4)
[*] 192.168.56.103:21 - USER: 331 Please specify the password.
[*] Exploit completed, but no session was created.
The analyst subsequently types the sessions command, which returns No active sessions.. This indicates that the initial payload was delivered, but the expected reverse or bind shell connection on port 6200 was never established or instantly died.
Packet-Level Diagnosis of the "500 OOPS" Exploitation Error
To truly operationalize this attack within a SOC context and troubleshoot the failed session, the analyst must understand how the exploit manifests at the network layer and why the daemon rejected the backdoor. Wireshark is utilized to dissect the traffic generated during the failed Metasploit execution.15
When analyzing the Packet Capture (PCAP) on the Ubuntu monitor, the analyst applies display filters to isolate the relevant traffic. Filtering for tcp.port == 21 reveals the initial FTP communication originating from Kali (192.168.56.10) to Metasploitable (192.168.56.103). By right-clicking the packet and selecting "Follow TCP Stream," the plaintext exchange becomes visible.17 The stream shows the attacker sending the exact trigger payload formulated by Metasploit: a Request containing the command USER followed by a random string appended with the trigger characters :) (specifically, USER test:) as captured in the Wireshark logs).12 The server responds normally with 331 Please specify the password..12 This is immediately followed by a Request containing the command PASS anything.12
It is at this exact moment the error occurs. Instead of seamlessly dropping into a root shell or acknowledging the password, the vsftpd daemon on Metasploitable returns a fatal error visible in the Wireshark capture: Response: 500 OOPS: priv_sock_get_result. Following this response, a series of red TCP Reset (RST) packets are transmitted, forcefully tearing down the connection.
This specific error fix requires a deep architectural explanation. The 500 OOPS: priv_sock_get_result error occurs because modern Linux kernels, or specific configurations of the Metasploitable instance, enforce strict privilege separation and utilize security modules like AppArmor or Secure Computing Mode (seccomp). When the malicious vsf_sysutil_extra() backdoor function attempts to call execl() to spawn /bin/sh on port 6200, the process lacks the necessary socket privileges or is blocked by an AppArmor profile restricting the vsftpd process from spawning arbitrary shells. Instead of successfully binding to port 6200, the daemon encounters an unrecoverable privilege violation and crashes, resulting in the 500 OOPS message sent back to the client, followed by the TCP RST.
To fix this error and allow the exploit to succeed in a laboratory environment, the SOC analyst must log directly into the Metasploitable virtual machine and temporarily disable or modify the AppArmor profiles or seccomp filters restricting the vsftpd daemon. Alternatively, modifying the vsftpd.conf file to run the service entirely as root without privilege separation allows the backdoor socket to bind successfully, generating the expected port 6200 traffic that the SOC monitor relies on for detection training.


Phase 4: Network Intrusion Detection and Protocol Analysis (Days 7-8)
Suricata Configuration and FTP Brute Force Execution
Days 7 and 8 pivot the laboratory focus toward network-based threat hunting using Suricata. Suricata is an advanced, high-performance Network Intrusion Detection System (NIDS), Intrusion Prevention System (IPS), and network security monitoring engine that operates by deeply inspecting network traffic against a vast, continuously updated database of signatures.18 In this laboratory phase, Suricata is deployed natively on the Ubuntu server, binding to the enp0s8 interface to monitor all traffic flowing toward the vulnerable endpoints.2
To rigorously test Suricata's detection capabilities, a brute-force attack against the FTP service is simulated using Hydra from the Kali Linux machine.4 Hydra is a highly optimized, parallelized network login cracker that supports numerous protocols.4 The analyst opens a terminal on Kali and executes the attack using the command hydra -l msfadmin -P /usr/share/wordlists/rockyou.txt ftp://192.168.56.103.4 This command instructs Hydra to target the FTP service on 192.168.56.103, repeatedly attempting to authenticate as the known user msfadmin by iterating through the comprehensive rockyou.txt password dictionary.4 A secondary execution utilizing fasttrack.txt successfully identifies the password msfadmin, resulting in a valid 230 Login successful response.
Decoding the Emerging Threats Suricata Ruleset
As the Hydra attack bombards the Metasploitable server, the SOC analyst monitors the live Suricata logs on the Ubuntu server by executing the command tail -f /var/log/suricata/fast.log. The terminal outputs continuous Priority 3 alerts, including SURICATA Applayer Mismatch protocol both directions and SURICATA TLS invalid record type due to Hydra's rapid, sometimes malformed connection tearing, alongside explicit brute-force alerts.
Suricata detects this specific brute-force activity through the implementation of community-sourced Emerging Threats (ET) rules. A rule designed to detect FTP brute-force attempts deeply analyzes the server's response codes.22 In the FTP protocol, a failed authentication attempt reliably returns a 530 status code indicating "Not logged in".23 The specific Suricata rule authored and parsed for this detection is structured as follows 24:
alert tcp any 21 -> any any (msg:"ET SCAN Potential FTP Brute-Force attempt"; flow:from_server,established; dsize:<100; content:"530 "; depth:4; pcre:"/530\s+(Login|User|Failed|Not)/smi"; classtype:unsuccessful-user; threshold: type threshold, track by_dst, count 5, seconds 300; sid:2002383; rev:12;)
This signature represents highly sophisticated protocol engineering. The alert tcp any 21 -> any any header specifies that the rule inspects TCP traffic originating from port 21 (the FTP server) heading to any destination IP and port.25 The flow:from_server,established keyword is a crucial optimization, ensuring the rule only wastes CPU cycles inspecting established connections originating from the server.25 The content:"530 " keyword looks for the specific ASCII byte sequence indicating a login failure, restricting the search to the first four bytes of the packet payload using depth:4 to maximize processing efficiency.22
Furthermore, a Perl Compatible Regular Expression (pcre) is utilized to match expected textual variations of the error message, such as "530 Login incorrect" or "530 User cannot log in", allowing a single rule to cover multiple FTP daemon implementations.25 Crucially, the threshold keyword tracks the destination IP and requires the exact event to occur 5 times within a 300-second temporal window before generating an alert.22 This thresholding mechanism is vital in a production SOC to prevent alert fatigue, ensuring the SIEM ignores isolated, accidental human login failures while reliably catching automated Hydra campaigns.26 When all conditions are met, Suricata writes the alert to the plaintext fast.log file and generates a highly detailed JSON payload in the eve.json file for subsequent SIEM ingestion.18
Suricata Rule Component	Analytical Purpose and Processing Impact
flow:from_server,established	Limits packet inspection to established TCP sessions originating from the FTP server, significantly reducing false positives and CPU overhead.
content:"530 "; depth:4	Performs a rapid string match for the FTP failure code exclusively within the first four bytes of the payload, preventing full-packet regex scans.
pcre:"/530\s+(Login|User|Failed|Not)/smi"	Applies complex pattern matching to catch variations in the daemon's failure response text, making the rule vendor-agnostic.
threshold: track by_dst, count 5, seconds 300	Correlates events over time, requiring 5 distinct failures targeting the same destination within a 5-minute window to trigger an alert.

Phase 5: Host-Based Intrusion Detection and EDR Analytics (Days 9-10)
Wazuh Architecture and SSH Attack Simulation
Following the network-centric approach of Suricata, Days 9 and 10 introduce Wazuh, a comprehensive Host-based Intrusion Detection System (HIDS) and Endpoint Detection and Response (EDR) platform.21 Wazuh agents are deployed natively on the endpoints (Metasploitable and Windows) to deeply monitor local system logs, file integrity, and configuration baselines, securely forwarding this telemetry to the centralized Wazuh manager hosted on the Ubuntu server.28
A primary exercise in this operational phase involves detecting Secure Shell (SSH) brute-force attacks against the Linux endpoint.21 Similar to the FTP scenario, Hydra is utilized from the Kali machine to bombard the SSH service on TCP port 22. The executed command is hydra -l msfadmin -P /usr/share/wordlists/rockyou.txt ssh://192.168.56.103.20 As Hydra aggressively opens and closes SSH sessions, the Metasploitable system begins buckling under the load. The Wazuh manager receives the continuous stream of authentication failure logs generated by the sshd daemon, extracted from /var/log/syslog or /var/log/auth.log via the local Wazuh agent.30
Deciphering the Wazuh Dashboard and Rule Correlation
The SOC analyst logs into the Wazuh Kibana dashboard hosted on the Ubuntu server to analyze the live security events. The dashboard displays a barrage of alerts associated with Agent 001 (metasploitable). The analyst observes a hierarchy of rules triggering simultaneously. Initial failed attempts trigger Level 5 alerts, including Rule 5503 (PAM: User login failed) and Rule 5760 (sshd: authentication failed). These rules map directly to the MITRE ATT&CK framework under T1021.004 (Remote Services: SSH) and T1110.001 (Brute Force: Password Guessing).
Wazuh's true analytical power lies in its hierarchical XML rule correlation engine. When a single failed login attempt occurs for an invalid user, Wazuh triggers Rule ID 5710.31 The definition for Rule 5710 utilizes simple regex matching for strings like "illegal user" or "invalid user" generated by the SSH daemon.31 This rule generates an alert at Level 5, representing a single, potentially benign failed attempt.31
To detect an actual automated brute-force campaign, Wazuh employs advanced rule chaining via the <if_matched_sid> XML tag.31 The dashboard reveals that the Level 5 alerts have been rapidly superseded by Level 10 critical alerts, specifically Rule 5551 (PAM: Multiple failed logins in a small period of time) and Rule 5712 (sshd: brute force trying to get access to the system. Non existent user). Rule ID 5712 is engineered specifically to identify sustained brute-force behavior by correlating the underlying Level 5 events.31
The structural definition of Wazuh Rule 5712 includes:
●	<if_matched_sid>5710</if_matched_sid>: This statement dictates that Rule 5712 only evaluates its logic if the base Rule 5710 has previously triggered within the system memory.31
●	frequency="8" timeframe="120": This establishes the temporal correlation threshold. The activity matching the base rule must occur exactly 8 times within a rolling 120-second window.31
●	<same_source_ip />: This critical parameter ensures that the 8 failed attempts originate from the exact same attacker IP address, differentiating a targeted attack from general internet background noise.31
●	<mitre><id>T1110</id></mitre>: This automatically tags the alert with the MITRE ATT&CK framework metadata, specifically mapping it to Technique T1110 (Brute Force) to aid in incident response categorization.30
When these specific temporal and spatial conditions are met, Wazuh suppresses the individual Level 5 alerts to prevent dashboard clutter and elevates the incident, generating a single, high-fidelity Level 10 alert.30 Furthermore, the Wazuh dashboard displays an unexpected Level 5 alert: Rule 40704 (Systemd: Service exited due to a failure). This provides immense situational awareness to the SOC analyst, indicating that Hydra's aggressive parallelized threading has actually caused the underlying authentication daemon or a related service on the Metasploitable victim to crash under the load, effectively turning a brute-force attack into an unintentional Denial of Service (DoS) event.

Phase 6: Web Application Security and SIEM Integration (Days 11-12)
Operating Burp Suite for Application Interception
Days 11 and 12 shift the threat-hunting focus toward the application layer, introducing Burp Suite to intercept and manipulate web traffic directed at the Metasploitable virtual machine. Metasploitable hosts several intentionally vulnerable web applications, such as Mutillidae and the Damn Vulnerable Web App (DVWA), operating on standard HTTP ports. Burp Suite operates as an intercepting proxy, positioned logically between the Kali Linux attacker's web browser and the target server.
The setup requires configuring the Kali Firefox browser to route all HTTP and HTTPS traffic through the local Burp Suite listener, typically bound to 127.0.0.1:8080. Once traffic is routed, the SOC analyst utilizes the Burp Proxy module to intercept an HTTP GET or POST request before it leaves the Kali machine. By manipulating the raw HTTP headers and parameter values in real-time, the analyst attempts to inject malicious payloads. For example, injecting ' OR 1=1 -- into a login field attempts a classic SQL Injection (SQLi), while injecting <script>alert('XSS')</script> into a comment field attempts a Cross-Site Scripting (XSS) attack. Burp Suite allows the analyst to meticulously craft these payloads and observe the raw HTTP response from the Metasploitable server, verifying if the application sanitizes the input or blindly executes the payload.
Detecting Web Application Attacks via Splunk SPL
While Burp Suite facilitates the attack, the true SOC objective is detecting these manipulations within the Splunk SIEM on the Ubuntu server. The web application logs (typically Apache access.log or generalized access_combined sourcetypes) are forwarded from Metasploitable to Splunk via a Universal Forwarder.
The SOC analyst authors complex Search Processing Language (SPL) queries to hunt for the payloads injected via Burp Suite. To detect the SQL Injection attempt, the analyst structures a query targeting the access logs, looking for common SQL keywords embedded within HTTP Uniform Resource Identifiers (URIs) 35:
sourcetype=access_combined | regex uri="(?i)(UNION|SELECT|INSERT|DROP|OR 1=1|--)" | stats count by clientip, uri, status | sort - count
This SPL query first filters the dataset to only include web access logs.36 It then utilizes the regex command to perform a case-insensitive (?i) search across the uri field for classic SQL injection syntax.36 The stats count command aggregates the results, showing exactly which clientip (the Kali machine) attempted the injection, the exact payload used, and the HTTP status code returned by the server (e.g., a 200 OK indicating a potentially successful injection, or a 500 Internal Server Error indicating a database crash).36
Similarly, detecting the Burp Suite XSS injection requires searching the access logs for HTML script tags. The SPL query is modified: sourcetype=access_combined | search uri="*<script>*" OR uri="*%3Cscript%3E*" | table _time, clientip, uri This query accounts for both plaintext script tags and URL-encoded variations (%3Cscript%3E) that Burp Suite might automatically encode during transmission, formatting the output into a readable table for incident response.37
Phase 7: System Resource and Process Monitoring (Days 13-14)
Terminal-Based Process Management: htop vs. vtop
The final two days of the laboratory roadmap pivot toward system administration and real-time incident response at the command-line level. When a Linux server (such as the Ubuntu SIEM or the Metasploitable target) exhibits performance degradation or suspected malicious activity—such as the daemon crashes observed during the Hydra attacks—terminal-based resource monitors are the first line of diagnostic defense.40 The traditional top command provides basic visibility, but modern SOC environments rely on more advanced variants: htop and vtop.41
Operational Mechanics of htop
htop is a highly mature, cross-platform interactive process viewer written in C, utilizing the ncurses library for its text-based user interface.41 Its low system overhead makes it ideal for analyzing compromised or heavily loaded servers without exacerbating resource exhaustion.43
Navigation and process manipulation in htop are heavily reliant on function keys and standardized shortcuts 45:
●	Navigation: Analysts use the directional arrow keys to scroll vertically through the process list and horizontally to view full execution paths.42
●	Sorting: Pressing P sorts the process list by CPU percentage, M sorts by memory consumption, and T sorts by execution time.45
●	Process Termination: If a malicious process (such as the vsftpd backdoor shell or a rogue script) is identified, the analyst highlights the process and presses F9.42 This invokes the kill menu on the left side of the screen, allowing the analyst to send a specific POSIX signal (e.g., 15 SIGTERM for a graceful exit or the forceful 9 SIGKILL) to terminate the process.40
●	Exiting: The utility is cleanly exited by pressing F10 or the q key.42
Operational Mechanics of vtop
While htop is an industry standard, vtop represents a modernized approach to terminal monitoring, prioritizing visual data representation over raw data density.41 Unlike the compiled C architecture of htop, vtop is written entirely in JavaScript and executes within the Node.js runtime environment.43 This fundamental architectural difference necessitates a specific installation procedure on the Ubuntu server, as it is not typically included in default repositories.50
The installation of vtop strictly requires the prior installation of Node.js and its package manager, npm (Node Package Manager).50 The requisite commands executed sequentially are:
1.	sudo apt install nodejs
2.	sudo apt install npm
3.	sudo npm install -g vtop 50
The -g flag in the npm command is critical; it ensures that vtop is installed globally across the system, making the executable binary accessible from any directory within the filesystem path.49
Once launched by simply typing vtop, the interface presents dynamic, ASCII-based real-time strip charts illustrating CPU and memory usage over time, offering a more intuitive visual baseline than the raw numerical data rows of htop.41 However, this visualization comes at a computational cost; vtop inherently consumes significantly more CPU overhead simply to render its own interface and query the system state compared to htop, making it less ideal for heavily compromised or starved systems.43
